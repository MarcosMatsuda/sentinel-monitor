01Overview

Visão do produto

SentinelMonitor é um dashboard B2B de múltiplas câmeras com pareamento permanente. Caso de uso: empresa de segurança instala câmeras na casa do cliente. Técnico abre URL no dispositivo da câmera (laptop / celular / no futuro IP cam), mostra código de 6 caracteres. Operador no QG digita o código no dashboard. Câmera fica permanentemente paireada — fechar/reabrir o dashboard reconecta automaticamente, sem novo pareamento.

Objetivos

  • Pareamento permanente — um código, uma vez. Nunca mais.
  • P2P privacy-first — vídeo + áudio fluem direto entre câmera e dashboard via WebRTC. Server só faz signaling.
  • Universal viewer — uma codebase Expo, roda em iOS / Android / Web.
  • Grid escalável — até 9 câmeras simultâneas em layout automático 1x1 / 2x2 / 3x3. Tap em tile = fullscreen.
  • Stateless server — sem DB no MVP. Clientes guardam bindings; server só gerencia presence + pairing codes efêmeros.
  • Phase 2 ready — arquitetura desenhada para AI cross-camera tracking sem refactor estrutural.

Constraints

  • MVP testa em casa via browsers em laptops. Phase 2+ aceita câmera de celular ou IP cam (RTSP bridge).
  • Máximo 9 câmeras por dashboard (limite de UI/CPU, não de protocol).
  • Sem auth no MVP — pairing code + UUID local servem como autenticação de fato.
  • Server fica em free tier (Render / Fly.io); TURN opcional (Metered.ca free tier).

2.1 Identity model

ConcernDecision
Camera identityUUID v4 do crypto.randomUUID(), persistido em localStorage["sm.cameraId"]
Dashboard identityUUID v4 em localStorage (web) / AsyncStorage (native), chave sm.dashboardId
Pairing code6 chars, alfabeto ABCDEFGHJKMNPQRSTUVWXYZ23456789 (32 chars, sem 0/I/L/O), TTL 5 min, single-use
Persistent bindingClient-side: dashboard guarda [{cameraId, label, addedAt}]; camera guarda pairedDashboards: string[]
Server stateIn-memory: presence map (peerId → socketId) + pairing codes ativos. Sem DB no MVP. Stateless em bindings — clientes são source of truth.
Trade-off conhecido Se a câmera limpar o localStorage (desinstalar app, modo anônimo, browser reset), o pareamento quebra. Câmera vira "novo dispositivo". Para MVP/portfolio aceita-se. Phase 3 (multi-tenant) adiciona DB + tokens assinados.

2.2 Topology

Mesh: 1 dashboard ↔ N câmeras = N peer connections independentes. Cada câmera pode estar paireada a M dashboards (M pequeno na prática — 1 ou 2 operadores por instalação). Cada peer connection compartilha o mesmo MediaStream da câmera, então adicionar um segundo dashboard custa só mais um SDP handshake, não mais uma captura.

Quando o número de viewers/câmeras crescer além do que mesh aguenta (Phase 3), migra para SFU central (mediasoup) que recebe stream uma vez por câmera e fanout para os viewers.

Cada app segue Clean Architecture (Uncle Bob / Hexagonal):

apps/<app>/src/
├── domain/           # Pure business logic. Zero framework deps.
│   ├── entities/
│   ├── repositories/    # Interface contracts (DI)
│   └── use-cases/
├── data/             # (server) Repository implementations
├── infrastructure/   # Concrete adapters (WebRTC, signaling, storage)
└── presentation/     # UI layer (components, screens, stores)

Layer rules: domain não importa nada das outras. data/infrastructure implementam interfaces de domain. presentation consome use cases via DI no composition root (main.ts ou _layout.tsx).

05Signaling protocol

Socket.IO event contract

EventDirectionPayloadPurpose
register-presenceclient → server{peerId, role}Camera ou dashboard se anuncia ao conectar
request-pairing-codecamera → server (ack){cameraId}Camera pede código de pareamento; ack devolve {code, expiresAt}
redeem-pairing-codedashboard → server (ack){code, dashboardId}Operador digita código; server resolve para cameraId atomicamente
pairing-redeemedserver → camera{cameraId, dashboardId}Push para a câmera saber quem é o novo dashboard parente
query-presencedashboard → server (ack){peerIds[]}Dashboard pergunta quais das suas câmeras estão online
subscribe-presencedashboard → server{peerIds[]}Dashboard quer ser notificado de mudanças nesse conjunto
presence-changeserver → dashboard{peerId, online}Push quando câmera subscrita conecta/desconecta
signalpeer → server → peer{fromPeerId, toPeerId, payload}Relay de SDP offer/answer + ICE candidates entre peers identificados por UUID
Por que sem rooms No baby-monitor a sala identifica o par. Aqui peers têm UUIDs estáveis, então identificamos diretamente. Server faz lookup toPeerId → socketId no presence map e relaya. Se o peer estiver offline, o signal é descartado silenciosamente.

5.1 DataChannel messages

Cada peer connection abre um DataChannel rotulado control. Mensagens trafegam diretamente entre os dois peers (não passam pelo server). Tipo discriminado:

type DataChannelMessage =
  | { type: 'video-meta'; width: number; height: number; ts: number }
  | { type: 'audio-mute-request'; muted: boolean; ts: number }   // dashboard → camera, advisory
  | { type: 'ping'; ts: number }
  | { type: 'pong'; ts: number }
  // Phase 2 forward-compat:
  | { type: 'detection-event'; entityType: 'person' | 'vehicle'; bbox: BoundingBox; ts: number };

6.1 Pairing (1ª vez)

Camera                              Server                          Dashboard
  load cameraId from localStorage     |                                |
  connect socket                      |                                |
  emit register-presence(camera, U)──►|                                |
                                      | presence[U] = socketA          |
  emit request-pairing-code(U)──────►|                                |
                                      | code=ABC123, ttl 5min          |
  ───ack {ABC123, exp}────────────|                                |
  show code on screen                 |                                |
                                                                      operator types ABC123
                                      |──redeem-pairing-code(ABC123,D)|
                                      | atomic redeem → cameraId=U  |
                                      |───ack {cameraId:U, dashId:D}──►|
  ───push pairing-redeemed{D}────|                                |
  save D in pairedDashboards          |                                | persist binding {U, label}
                                                                      subscribe-presence([U])
                                                                      emit signal: offer to U

6.2 Reconnect (toda vez que dashboard reabre)

Dashboard                              Server
  AsyncStorage → bindings=[U1,U2,U3]    |
  localStorage → dashboardId=D          |
  connect socket                        |
  emit register-presence(dashboard, D)──►|
  emit query-presence([U1,U2,U3])──────►|
                                        | check presence map
  ───ack {online:[U1, U3]}─────────────|
  emit subscribe-presence([U1,U2,U3])──►|
  for each online peer: ConnectToCameraUseCase()
                                        | route signals as before
  later, U2 reconnects:
                                        |── register-presence(camera, U2) from socketB
  ───presence-change{U2, online:true}───|
  ConnectToCameraUseCase(U2)

08Test strategy

Stack por app

AppFrameworkCoverage target
serverJest + ts-jest + supertest90% domain + use cases, 80% socket handler integration
cameraVitest + happy-dom + fake-rtc helper80% domain + use cases + signaling client
viewerJest + ts-jest (lib code only, não componentes RN no MVP)80% domain + use cases + stores

Regras

  • Cada acceptance criterion da issue do GitHub mapeia para 1+ teste
  • Mock todos os ports/repos externos. Nunca mocka a coisa que está sendo testada.
  • clearAllMocks em beforeEach
  • Sem testes que dependem de execution order
  • Testes que sintetizam ou consomem vídeo usam fakes (fake-rtc.ts, fake-audio-context.ts) reaproveitados do baby-monitor
  • Skip component tests no MVP — entram em fase posterior se valer

11Risks & mitigations

RiskImpactProbabilityMitigation
react-native-webrtc + Expo Universal High Medium Custom dev client (não Expo Go). Pin react-native-webrtc@124.x. Web build é o inner-loop dev path; native validado por PR.
Platform.select footguns Medium Medium File-extension resolution (.web.ts/.native.ts) ao invés de Platform.OS em runtime. CI assert garantindo que react-native-webrtc não entra no bundle web.
Mesh 1x9 CPU/banda em mobile Medium High Cap default 320x240@15fps via webrtc-config. Auto-aplicar preset "low" quando grid ≥ 5. Pausar peers não-visíveis em fullscreen. SFU em Phase 3.
Race em pairing redemption Low Low Atomic delete-and-return no in-memory repo. Single-use codes by design. Retorna PairingErrorDto tipado em miss.
Presence drift após server restart Medium Medium Clientes re-registram em todo socket reconnect. Dashboards re-query presence. Signal route descarta silenciosamente em recipient offline. SLA: reconverge em 1 ciclo.
localStorage reset quebra binding High Low Documentado como limitação conhecida. Phase 3 (multi-tenant) adiciona DB + tokens assinados que sobrevivem ao reset local.

12Future phases (informs MVP today)

v2 - AI cross-camera tracking

Operador marca pessoa/veículo como suspeito em Cam 1; quando a mesma entidade aparece em qualquer outra câmera, dashboard alerta. Computer-vision territory: object detection + person/vehicle re-identification + ALPR.

Capacidade Phase 2Hook arquitetural no MVP
Object detection per-camera (YOLO / RT-DETR ou cloud Vision API) Camera publisher expõe i-frame-analyzer.repository.ts — MVP ship NoOpFrameAnalyzer; v2 troca por inference real
Watchlist de embeddings cross-camera Server ganha canal watchlist.broadcast. MVP só deixa o canal existir como no-op
Detection events (camera → dashboard) Reuso do DataChannel: {type: 'detection-event', bbox, ts} já está no union type do MVP
Cross-camera correlation (ReID, plate match) Server ganha use case CorrelateDetectionsUseCase. MVP só precisa do event bus existir
Alert fanout para dashboard Reuso do presence/subscription: server emite cross-camera-alert

Tech direction (decidido em v2): provavelmente on-device inference para object detection (TensorFlow.js / WebGPU no browser da câmera, ou react-native-fast-tflite em mobile cameras), cloud inference para ReID (Anthropic API com vision, ou hosted embedding model). Bounding boxes fluem para dashboard como overlay; embeddings fluem para server para cross-camera match.

v3 - Scale & produto

  • Multi-tenant + auth (Clerk/Supabase)
  • Recording contínua / triggered → Cloudflare R2
  • SFU central (mediasoup) quando mesh não aguentar
  • Motion zones configuráveis
  • RTSP bridge para IP cameras tradicionais (go2rtc / Janus)
  • Analytics + relatórios

13Dev workflow

Branching

  • main — locked. Só recebe o release v1.0.0 quando o produto estiver feature-complete.
  • develop — default integration branch. Todo PR de feature aponta para develop.
  • feat/issue-NN-*, fix/issue-NN-*, chore/issue-NN-*, test/issue-NN-* — uma branch por issue.

Branch protection

  • main: PR + 1 human approval + CI verde + linear history
  • develop: CI verde (audit + lint + typecheck + tests) + code-reviewer agent verdict not-failing

Issue-driven autonomous loop

Cada PR nasce de uma GitHub issue com acceptance criteria objetivos. Loop por issue:

  1. gh issue view → ler critérios
  2. gh issue edit --add-label "status:in-progress"
  3. Branch from develop, implement (Clean Arch + SOLID)
  4. Escrever tests — um por acceptance criterion
  5. Local: pnpm typecheck && pnpm test:unit
  6. Commit conventional + Closes #NN
  7. Push, gh pr create --base develop
  8. Dispatch code-reviewer agent
  9. Se [CRITICAL]/[MAJOR]: corrigir (max 2 retries) ou escalar
  10. CI verde → gh pr merge --squash --delete-branch
  11. gh issue close NN → próxima issue

Release v1.0.0

Quando produto estiver feature-complete e validado em devices reais (iPad + browsers + native viewer + camera publisher em redes diferentes com TURN), abrir PR develop → main intitulado release: v1.0.0 com changelog gerado dos commits. Aprovação humana obrigatória.
